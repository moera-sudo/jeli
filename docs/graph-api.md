# Граф: как это устроено на бэкенде (для фронтенда)

Этот файл описывает фичу `graph` (`jeli-server/src/features/graph/`) — как устроены данные, какие есть эндпоинты, как обходится граф и на что обратить внимание при разработке фронтенда. Дополняет `api-overview.md`, `matching-algorhitm.md` и `graph-data-structure.md` — здесь описана уже РЕАЛЬНО реализованная версия (некоторые детали успели измениться по ходу разработки относительно исходных доков).

## 1. Сущности

Отдельной сущности "Граф" в БД нет. Граф пользователя — это просто множество узлов `Person`, у которых `owner_user_id` равен его id. Всё остальное — рёбра между узлами.

| Таблица | Что это | Ключевые поля |
|---|---|---|
| `persons` | Узел графа (человек — живой или умерший, зарегистрированный или нет) | `id`, `owner_user_id`, `linked_user_id` (nullable), `origin_label`, `full_name`, `gender`, `is_alive`, `birth_year_value`/`precision`, `death_year_value`/`precision`, `death_context`, `birth_country`, `birth_region`, `ru`/`tribe`/`zhuz`, `invite_code` (nullable) |
| `relationships` | Ребро: кровное родство или брак | `from_person_id`, `to_person_id`, `type` (`child_of`\|`spouse_of`), `marriage_year`, `marriage_end_reason` |
| `graph_links` | Дополнительный "мост" между кластерами — только для агрегации мэтчей и отображения, не для генеалогии | `person_a_id`, `person_b_id`, `link_type` (`marriage`\|`match_confirmed`) |
| `relationship_proposals` | Предложение брака между узлами | `proposer_user_id`, `person_a_id`, `person_b_id`, `status` (`pending`\|`confirmed`\|`rejected`) |
| `match_candidates` | Кандидат в мэтчи — таблица создана сейчас, но заполняется алгоритмом мэтчинга только в следующем этапе разработки | `person_a_id`, `person_b_id`, `score`, `status`, `evidence` |
| `graph_collaborators` | Делегирование прав редактирования графа | `graph_owner_id`, `collaborator_user_id` |

### 1.1. Важные конвенции

- **`child_of` направлен от ребёнка к родителю**: `from_person_id` = ребёнок, `to_person_id` = родитель. Не наоборот.
- **`spouse_of`** не участвует в поиске кровного родства — только для отображения объединённого семейного вида и построения household-графа.
- **У узла максимум 2 родителя** (`child_of`-рёбер "от себя" не больше двух).
- **`generation` нигде не хранится** — считается на лету при каждом запросе (глубина от `focus`-узла в конкретном ответе).
- **`origin_label`** — служебная метка кластера (UUID). Все узлы одного изначально несвязанного куска дерева имеют одинаковый `origin_label`. При подтверждении брака между разными владельцами весь кластер одной стороны перекрашивается в `origin_label` другой (union-find). Фронту эта метка не нужна напрямую, но объясняет, почему у только что объединённых деревьев вдруг "совпадают" внутренние группировки.
- **`is_registered`** (в ответах графа) = `linked_user_id IS NOT NULL`. **`can_chat`** считается сервером (жив + зарегистрирован + это не сам смотрящий) — фронт просто рисует кнопку по этому флагу, никакой своей логики.
- **`can_edit`** (в `PersonDetail`) — можно ли ТЕКУЩЕМУ пользователю редактировать/удалять этот конкретный узел. Считается сервером (владелец графа, коллаборатор, или сам живой человек, если узел привязан к нему). Используйте это поле, чтобы решать, показывать ли кнопки "редактировать"/"удалить"/"сделать коллаборатором" на карточке узла — не дублируйте логику на фронте.

## 2. У пользователя может быть только одно дерево

Никакого автосоздания дерева при регистрации не происходит. Есть два способа получить свой узел:

- **`POST /api/graph/create`** — создаёт новый корневой узел (используется `gender` из профиля пользователя — если он ещё не заполнен через `/api/users/profile/edit` или `/api/users/create`, вернётся 409 с понятным сообщением). Обычный путь для человека, который начинает своё дерево с нуля.
- **`POST /api/graph/join`** (body: `{"invite_code": "..."}`) — присоединение к уже существующему узлу, который кто-то из родственников завёл заранее и поделился кодом. 404, если код неверный или узел уже кем-то занят.

Оба эндпоинта вернут 409 (`AlreadyHasPersonError`), если у пользователя уже есть привязанный узел (`GET /api/persons/me` не 404).

Также `graph_invite_code` можно передать прямо в `POST /api/auth/register` — это best-effort попытка присоединиться сразу при регистрации (без ошибки, если код неверный — тогда просто продолжаете без дерева и вызываете `/graph/create` или `/graph/join` позже вручную).

**При успешном присоединении** (и через `/graph/join`, и через код в регистрации) в профиль пользователя (`Users`) копируются поля `gender`, `ru`, `tribe`, `zhuz`, `birth_country` из узла — но только те, что там ещё не заполнены. Если пользователь уже сам указал что-то в профиле — не перезаписывается.

## 3. Код приглашения (`invite_code`)

Один и тот же код используется для ДВУХ разных действий:

1. **Присоединение аккаунта к узлу** (`POST /api/graph/join`, а также `graph_invite_code` в `/api/auth/register`) — "это я, вот мой узел".
2. **Таргет предложения брака** (`target_invite_code` в `POST /api/marriage-proposals`) — "вот код узла, с которым я хочу связать браком человека из своего дерева".

Формат: **8 символов, Crockford Base32** (`0-9`, `A-Z` без `I`/`L`/`O`/`U` — не путается на слух и при ручном вводе). Коды не привязаны к ссылкам/QR — предполагается, что ими делятся текстом через любой мессенджер.

Код генерирует владелец узла: **`POST /api/persons/{id}/invite-code`** → `{"invite_code": "ABCD1234"}`.

Код **не одноразовый** и не обнуляется после использования — можно и присоединиться по нему, и потом ещё раз использовать его же для брака (или наоборот). Эксклюзивность присоединения (чтобы два разных человека не заявили себя одним и тем же узлом) обеспечивается проверкой `linked_user_id IS NULL` в момент клейма, а не самим кодом.

**Важно для фронта**: раз глобального поиска чужих графов нет и не планируется, ЕДИНСТВЕННЫЙ способ связать два независимо созданных дерева (по коду для брака) или привязать живого человека к уже существующему узлу — это код. Продумайте UI для "поделиться кодом" (кнопка "скопировать код" на карточке узла + инструкция передать его вне приложения) и форму "ввести код" в двух местах: при регистрации (необязательное поле) и отдельно на экране "Присоединиться к дереву" / "Предложить брак".

## 4. Права редактирования

Три уровня доступа к узлу:

1. **Владелец** (`Person.owner_user_id == текущий пользователь`) — полный доступ.
2. **Коллаборатор** — владелец делегировал права редактирования ВСЕГО своего графа другому человеку (`POST /api/graph/collaborators`, body: `{"person_id": "..."}`). Коллаборатором можно сделать **только уже живой зарегистрированный узел своего же графа** (у узла должен быть `linked_user_id`) — то есть сначала родственник должен зарегистрироваться и присоединиться по коду, и только потом ему можно выдать права. Управлять коллабораторами может только сам владелец (не другой коллаборатор).
3. **Сам живой человек** — если узел привязан к текущему пользователю (`linked_user_id == он сам`), он тоже может его редактировать (`PATCH /persons/{id}`), даже если не является владельцем графа. Это ДОПОЛНИТЕЛЬНОЕ право, не заменяет права владельца.

**Чтение графа полностью открыто** любому авторизованному пользователю — приватности на уровне "нельзя посмотреть чужой узел" нет, это заложено в продукт (поиск родственников по крупицам данных). Закрыты только МУТАЦИИ.

Рекомендуемый UI-флоу для коллабораторов (уже обсуждали в разработке): открываете свой граф, кликаете на узел живого родственника → всплывает мини-карточка с его данными и кнопками "Сделать коллаборатором" / "Редактировать" / "Удалить" — видимость кнопок берите из `PersonDetail.can_edit` и `is_registered`.

## 5. Обходы графа — три разных эндпоинта, разное поведение

| Эндпоинт | Что возвращает | Ограничение | Супруги |
|---|---|---|---|
| `GET /api/graph?focus={id}&depth={n}` | Граф вокруг узла для React Flow (`persons[]`/`relationships[]`) | По `depth` поколений вверх/вниз (1–8, по умолчанию 3) | См. ниже |
| `GET /api/persons/{id}/bloodline` | Строго прямая линия предков/потомков | Без ограничения | Не участвуют вообще |
| `GET /api/persons/{id}/household-graph` | То же, что `GET /graph`, но без ограничения по глубине | Без ограничения | См. ниже |

`GET /graph` и `household-graph` используют один и тот же волновой обход (двунаправленный: вверх к родителям, вниз ко всем детям — поэтому сиблинги/племянники тоже попадают в граф, не только прямая линия).

### 5.1. Правило для супругов — принципиально важно для фронта

- **Действующий брак** (`marriage_end_reason` пуст) — обход ПРОДОЛЖАЕТСЯ через супруга: подтягивается вся его линия предков/потомков. Это то, что "сливает" два независимо созданных дерева после брака между ними — ребёнок от такого брака увидит бабушек-дедушек по обеим линиям.
- **Расторгнутый брак** (`marriage_end_reason` = `divorce`/`widowed`) — супруг показывается только сам, листом, БЕЗ его семьи. Общий ребёнок продолжает показываться нормально (это его кровный родитель, никуда не девается).
- **`match_confirmed`**-мосты (после подтверждения мэтча алгоритмом — Этап 4) — всегда один хоп, без рекурсии, независимо ни от чего. Рендерятся отдельным типом ребра `"match_confirmed"` в `relationships[]` (у него нет обычного `Relationship`, только `GraphLink`).

Типы рёбер в ответе (`relationships[].type`): `"child_of"`, `"spouse_of"`, `"match_confirmed"`.

### 5.2. Формат ответа (`GraphResponse`)

```json
{
  "focus_person_id": "uuid",
  "persons": [
    {
      "id": "uuid",
      "full_name": "Асель Ержанова",
      "gender": "female",
      "avatar_url": "https://...",
      "generation": 0,
      "birth_year": 1990,
      "death_year": null,
      "is_alive": true,
      "is_registered": true,
      "can_chat": false,
      "has_more_ancestors": true
    }
  ],
  "relationships": [
    { "id": "uuid", "from_person_id": "uuid", "to_person_id": "uuid", "type": "child_of" }
  ]
}
```

`has_more_ancestors` — есть ли у узла ещё родители за пределами возвращённого среза (только для `GET /graph` с ограничением по `depth` — используйте для кнопки "показать ещё поколение" и повторного запроса с бОльшим `depth`).

### 5.3. Подробная карточка узла

`GET /api/persons/{id}` — тяжёлая версия (`PersonDetail`): все поля + `relation_to_viewer` (упрощённая структурная формулировка вида "Предок (2 поколения)"/"Потомок (1 поколение)"/"Супруг(а)"/"Это вы" — полноценная казахская терминология родства (нағашы/аға/іні/жиен) пока не реализована), `chat_thread_id` (пока всегда `null` — фича `messenger` ещё не готова, кнопка "написать" должна быть неактивна), `top_matches` (пока всегда пустой список — алгоритм мэтчинга ещё не реализован), `can_edit`.

## 6. Редактирование графа

### 6.1. Добавление человека

`POST /api/persons` — атомарно создаёт узел, опционально сразу со связью к существующему (`relation`):

```json
{
  "full_name": "...", "gender": "male", "...": "...",
  "relation": { "to_person_id": "uuid", "type": "parent", "marriage_year": null, "marriage_end_reason": null }
}
```

`relation.type` читается **относительно `to_person_id`**: `"parent"` — новый человек станет родителем указанного узла, `"child"` — ребёнком, `"spouse"` — супругом. Для `"spouse"`, если `to_person_id` принадлежит ДРУГОМУ владельцу — вместо мгновенной связи создастся предложение брака (см. ниже).

### 6.2. Кровные связи между уже существующими узлами

`POST /api/relationships` (только `child_of`) — оба узла должны редактироваться вами, **либо** (если узлы уже в одном объединённом браком/мэтчем кластере) достаточно прав хотя бы на одну сторону — это позволяет добавлять детей смешанного брака без нового подтверждения на каждого ребёнка, раз доверие между двумя семьями уже установлено на уровне брака.

`DELETE /api/relationships/{id}` — удалить ребро целиком (для ошибочно созданной связи).

### 6.3. Вставка человека между двумя узлами

`POST /api/persons/insert-between` (`parent_id` + `child_id` + данные нового человека) — если между двумя УЖЕ существующими напрямую связанными узлами обнаружилось пропущенное поколение, эта операция вставляет нового человека между ними одной атомарной транзакцией (старое ребро удаляется, создаются два новых через нового человека). Специально сделано, чтобы не приходилось удалять/пересоздавать узлы вручную — обычное удаление каскадное и стёрло бы всех потомков ниже по линии.

### 6.4. Удаление/отвязка узла

`DELETE /api/persons/{id}?new_owner_user_id={uuid?}` — **ключевой нюанс**: если у узла есть `linked_user_id` (привязан живой аккаунт — свой или чужой), запись **никогда не удаляется физически** — только отвязывается (`linked_user_id = null`), все данные и родственные связи остаются. Реально удаляются (каскадно, вместе со связями) только узлы БЕЗ привязанного аккаунта.

- Если это **чужой** живой узел — просто отвязывается, узел остаётся видимым в графе как обычный (уже неживой в смысле аккаунта) человек. Отвязанному пользователю пока ничего не "прилетает" (полноценные уведомления — будущий этап), но технически это уже заложено.
- Если это **свой собственный** узел (`linked_user_id == вы`) — сначала проверяются кандидаты на передачу владения остальным графом: `GET /api/graph/successor-candidates` (живые зарегистрированные родственники в вашем графе + существующие коллабораторы). Если кандидаты есть, а `new_owner_user_id` не передан — 409 (`SuccessorRequiredError`): фронт должен показать пикер "кому передать управление графом" и повторить запрос с выбранным `new_owner_user_id`. Если кандидатов нет — отвязка происходит сразу, без вопросов.

## 7. Брак между двумя независимо созданными деревьями

Ключевой сценарий продукта: два человека независимо завели свои деревья, потом узнали, что породнились браком, и хотят объединить деревья для отображения (без физического слияния данных — оба дерева остаются отдельными, просто "видны" друг другу через ребро).

Флоу:
1. Один из владельцев генерирует код для СВОЕГО узла (`POST /persons/{id}/invite-code`) и передаёт его второму вне приложения.
2. Второй вызывает `POST /api/marriage-proposals`: `{"person_a_id": "<свой узел>", "target_invite_code": "<полученный код>", "marriage_year": 2015}`.
3. Если оба узла оказались бы редактируемы одним и тем же пользователем — брак подтверждается мгновенно (`status: "confirmed"`). Обычно (разные владельцы) — создаётся `status: "pending"`.
4. Второй владелец видит предложение через `GET /api/marriage-proposals` (входящие/исходящие) и подтверждает: `POST /marriage-proposals/{id}/confirm`, либо отклоняет: `.../reject`.
5. После подтверждения — оба дерева видны друг другу через `spouse_of`-ребро (см. правило про действующий брак в разделе 5.1).

Повторное предложение между той же парой, пока есть `pending` — 409 (`DuplicateProposalError`).

### 7.1. Развод/вдовство

`PATCH /api/relationships/{id}` (только для `spouse_of`): `{"marriage_end_reason": "divorce", "marriage_year": ...}` — **не удаляет** ребро, просто помечает его завершённым. Это важно: удаление стёрло бы факт брака и год навсегда, а `PATCH` сохраняет историю. Общий ребёнок остаётся связан с обоими родителями как обычно; в графе бывший супруг после этого показывается только сам, без своей более широкой семьи (см. 5.1).

## 8. Мэтчи — пока заглушка

`GET /api/persons/{id}/matches`, `GET /api/users/{id}/matches`, `POST /api/matches/{id}/confirm|reject` — все эндпоинты уже работают и корректно проверяют права/404, но таблица `match_candidates` пока пустая — сам алгоритм мэтчинга (5-этапный пайплайн из `matching-algorhitm.md`) будет реализован отдельным этапом. Можно уже сейчас верстать UI под эти эндпоинты (список карточек с `score`/`status`/`evidence`, кнопки confirm/reject) — как только алгоритм заработает, данные появятся без изменения контракта.

## 9. Рекомендации для фронтенда

- **Ленивая подгрузка графа**: используйте `GET /graph?focus&depth=3` по умолчанию, `has_more_ancestors` — сигнал для кнопки "показать ещё поколение" (повторный запрос с бОльшим `depth` от того же `focus`, либо смените `focus` на нужный узел).
- **Не дублируйте права/видимость на фронте** — `can_edit`, `is_registered`, `can_chat` всегда приходят готовыми с сервера, ориентируйтесь строго на них.
- **Мини-карточка узла** при клике: дёргайте `GET /persons/{id}` (тяжёлая версия с `relation_to_viewer`), показывайте кнопки по `can_edit`/`is_registered`.
- **Код приглашения** — предусмотрите UI и для генерации (кнопка на карточке своего узла), и для ввода (при регистрации — необязательное поле; отдельно — экраны "Присоединиться" и "Предложить брак"). Это единственный способ связать независимые деревья.
- **Self-delete всегда идёт через пикер**: перед вызовом `DELETE /persons/{id}` для СВОЕГО узла — сначала дёрните `GET /graph/successor-candidates`; если список непустой, покажите выбор преемника и передайте `new_owner_user_id`, иначе получите 409.
- **`chat_thread_id` и `top_matches` пока не реальны** (`null`/`[]`) — не блокируйте разработку UI на них, но кнопка "написать" должна быть задизейблена, пока `messenger` не реализован.
- Сообщения об ошибках (`detail` в теле ответа) — всегда на английском, независимо от того, что Swagger-документация и данные графа (имена, гео) на русском/казахском.
