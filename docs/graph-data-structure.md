## Главный принцип: плоские списки, а не вложенный JSON

Соблазн отдать дерево вложенно (`{person, children: [...]}`) — и это сломается сразу. У человека два родителя: при вложенности его придётся либо дублировать в двух ветках, либо уйти в бесконечную рекурсию на общих предках. Это ровно то, о чём мы говорили: DAG во вложенный JSON не укладывается.

Поэтому отдавай **два плоских массива** — они ложатся на React Flow один в один:

```
persons[]        → nodes[]
relationships[]  → edges[]
```

Фронт из них выводит союзы и раскладку. Бэкенд про отрисовку не знает вообще ничего.

## Эндпоинт 1 — граф

```
GET /graph?focus={person_id}&depth=3
```

```json
{
  "focus_person_id": "uuid-me",
  "persons": [
    {
      "id": "uuid-asan",
      "full_name": "Асан Сейітов",
      "gender": "male",
      "generation": 2,
      "avatar_url": "https://.../asan.jpg",
      "birth_year": 1932,
      "death_year": 1998,
      "is_alive": false,
      "is_registered": false,
      "can_chat": false,
      "has_more_ancestors": true
    }
  ],
  "relationships": [
    { "id": "r1", "from_person_id": "uuid-me", "to_person_id": "uuid-asan", "type": "parent_of" },
    { "id": "r2", "from_person_id": "uuid-asan", "to_person_id": "uuid-aigul", "type": "spouse_of" }
  ]
}
```

Полей мало намеренно — это «лёгкая» версия человека. Разберу те, которых нет в твоей текущей спеке, но без которых фронт не соберётся:

|Поле|Зачем|
|---|---|
|`generation`|**Критично.** Это координата Y. Фронт сам её не посчитает: при смешанных браках пришлось бы обходить весь граф и всё равно ошибиться. У тебя она уже есть из recursive CTE — просто отдай.|
|`is_alive`|Не выводится из `death_year`: человек может быть давно умершим с неизвестным годом смерти. Нужен явный флаг.|
|`is_registered`|Второе состояние карточки — «Пригласить».|
|`can_chat`|Сервер решает, можно ли писать (жив + зарегистрирован + приватность + не заблокирован). Фронт эту логику не повторяет, просто рисует кнопку.|
|`has_more_ancestors`|Без него фронт не знает, показывать ли кнопку «показать ещё поколение» — обрезал ты ветку по `depth` или там реально конец.|

`depth` даёт ленивую подгрузку: грузишь 3 поколения, по кнопке дозапрашиваешь глубже. Иначе на большом краудсорсинговом графе первый рендер положит браузер.

## Эндпоинт 2 — карточка

Возвращает то же самое **плюс всё тяжёлое**: `birth_country`, `birth_region`, `ru` / `tribe` / `zhuz`, `ethnic_source`, `source_type`, `attachments[]`, `confirmation_count`, `death_context`, точности дат.

Плюс три вычисляемых поля:

```json
{
  "relation_to_viewer": "прадед по отцовской линии",
  "chat_thread_id": "uuid | null",
  "top_matches": [
    { "id": "m1", "score": 0.82, "status": "high_confidence", "evidence": { "chain_length": 3, "sibling_confirmed": true } }
  ]
}
```

`relation_to_viewer` считай **только на сервере**. Казахская терминология родства нетривиальна (нағашы, аға, іні, жиен) — это доменная логика, ей не место на фронте. У тебя уже есть `relation_path_to_viewer` в `MatchCandidate`, переиспользуй.

`top_matches` я бы **вложила прямо сюда** (топ-3), а не делала отдельный запрос. Минус — кэш матчей и профиля живут по одному ключу; плюс — модалка открывается мгновенно, одним запросом. На 4 дня плюс важнее.

**Почему два эндпоинта, а не один.** Если слать полные данные на все узлы, на 200 узлах ответ раздувается в разы, а пользователь откроет одну карточку. Лёгкий граф + подробности по клику — стандартное разделение, и его легко защитить.

## Мутации и чат

```
POST /persons          — добавить родственника (атомарно, см. ниже)
GET  /threads/{id}/messages
POST /threads/{id}/messages
```

По добавлению — важная деталь. В твоей текущей спеке это два вызова: `POST /persons`, потом `POST /relationships`. Если второй упадёт, в базе останется человек, висящий в пустоте, и он попадёт в матчинг как мусор. Сделай **одним атомарным запросом**:

```json
POST /persons
{
  "full_name": "Бекнұр Асанов",
  "gender": "male",
  "relation": { "to_person_id": "uuid-asan", "type": "child" }
}
```

Одна транзакция — либо человек со связью, либо ничего. И фронту проще: одна кнопка = один запрос = одна инвалидация кэша.

## Итого

**Для графа и карточки — 2 эндпоинта.** Полный сценарий (с добавлением и чатом) — **5**.

Кэш в TanStack Query: `['graph', focusId, depth]` и `['person', id]`. После `POST /persons` инвалидируешь только `['graph', ...]` — дерево перестроится само.

## Два предупреждения

**1. У тебя `parent_of` названо задом наперёд.** По спеке `from` = ребёнок, `to` = родитель. Но «A parent_of B» читается как «A — родитель B», то есть ровно наоборот. Кто-нибудь в команде это перепутает и построит дерево вверх ногами — почти гарантированно. Переименуй в `child_of`, раз направление именно такое. Дёшево сейчас, больно потом.

**2. `can_chat` считает сервер, не клиент.** Соблазн написать на фронте `is_alive && is_registered` — но у тебя же есть приватность: живые 1–2 поколения скрыты, плюс блокировки. Права доступа на клиенте — это дыра, и жюри по критерию «безопасность» такое ловит. Сервер отдаёт готовый флаг.
