**«Дерево»** — это строгий термин с одним правилом: у каждого узла ровно один родитель.
Если строить через react-d3-tree или react-organizational-chart получится:

```
Асан        Айгүл
   \         /
    \       /
   Бекнур, Динара
```

Но у человека два родителя, а дерево по определению допускает одного. Плюс в родословной ветви сходятся обратно у общих предков.

Поэтому для рисовки семейного древа используется *DAG* — направленный ациклический граф:
- направленный — стрелки идут только в одну сторону, от ребёнка к родителю;
- ациклический — циклов нет, потому что нельзя быть предком самому себе.

#### Во избежание каши будет следующий рендеринг:

```
Асан ──┐
       ●          ← узел-союз (невидимая точка)
Айгүл ─┘
       │
   ┌───┼───┐
  Бек Дина Ерлан
```
Так не будет пересечений линий, и понятно кто кому родственник + через узел все кто ниже понятно, что братья и сестры. Узел — не костыль. Он отражает, что ребёнок принадлежит паре, а не одному человеку.

### Y координата получается с бэкенда (поколения), а X считается библиотекой библиотеке (Dagre)

Dagre не знает про поколения и из-за узлов-союзов добавляет лишний слой. А номер поколения у нас уже есть с бэкенда, он точный. Поэтому X берём у dagre — он умеет минимизировать пересечения линий, — а Y считаем сами и гарантируем ровные поколения.

Важное архитектурное правило: **координаты не пишутся в БД никогда.** Они всегда выводятся из структуры графа. Отсюда:

```jsx
<ReactFlow
  nodesDraggable={false}      // иначе юзер руками сломает поколения
  nodesConnectable={false}    // связи создаются только через форму, не перетаскиванием
  onlyRenderVisibleElements   // производительность на больших деревьях
  fitView
/>
```

Это заодно защищает данные: пользователь не сможет мышкой создать невалидное родство в обход проверок.
## Добавление родственника «в нужное место»

Не свободное перетаскивание, а контекстные кнопки на карточке — через встроенный `<NodeToolbar>`:

`+ Отец` · `+ Мать` · `+ Супруг(а)` · `+ Ребёнок` · `+ Брат/сестра`

Каждая кнопка знает свой контекст и разворачивается в две ровно твои ручки API:

```
POST /persons          → создать узел
POST /relationships    → parent_of / spouse_of с нужным направлением
→ invalidateQueries → рефетч → пересборка союзов → релэйаут
```

Дерево перестраивается само. 
## Три состояния карточки 

|Состояние|Вид|Действия|
|---|---|---|
|Живой + зарегистрирован|цветное фото, зелёная точка|Профиль + **Чат**|
|Живой, не зарегистрирован|фото, серая точка|Профиль + **Пригласить**|
|Умерший|сепия/ч-б, годы жизни|Только профиль (мемориал)|

Второе состояние — не мелочь, а **петля роста**: приглашение родственника напрямую кормит краудсорсинг и сетевой эффект, про который ты пишешь в слайде про устойчивость. Стоит показать это жюри как продуманную механику, а не случайность.

## Стек

- **Vite + React + TypeScript**
- **`@xyflow/react`** (v12) — граф
- **`@dagrejs/dagre`** — X-раскладка (поддерживаемый форк, оригинальный dagre заброшен)
- **TanStack Query** — серверное состояние; его инвалидация кэша идеально ложится на твой событийный пересчёт матчей на бэке
- **Zustand** — UI-состояние (выбранный человек, открытая модалка). Маленький, объяснить легко
- **framer-motion** (опционально) — плавная анимация при перестройке дерева. Дёшево, а в демо смотрится сильно

Не рендерь весь краудсорсинговый граф — он потенциально огромный. Эго-центрично, от текущего человека: твой `/persons/{id}/household-graph` с лимитом глубины, дальше кнопка «показать ещё поколение». Это и производительность, и приватность (у тебя живые 1–2 поколения и так скрыты из авто-матчинга).
